# 基础
- 用中文对话，用中文编写注释和文档
- 每个包含业务逻辑的代码文件，都要有一个同名的 .md 文件记录需求、实现细节、功能验证的具体计划或者操作步骤方法。如果代码文件缺少对应 `.md` 文件，就创建一个。每次修改代码，一定要同步更新 md 文件
- 实现完功能，必须按照功能验证计划或步骤，执行功能验证。执行验证时如果需要启动dev服务器，不要征求我的同意，直接执行。
- 实现功能时，优先尽快实现最小闭环，走通核心流程。先做核心功能，再考虑非核心功能。
- 除非我主动调用，否则不要使用 /superpowers:brainstorming
- 大部分常见 bash 命令不要征求我的允许，直接运行。
- 所有接口和后端逻辑都要放到 hono workers/api-worker 里面 里面
- 排查 bug 时要打印尽可能多和详细的日志。
- 添加页面功能或者修复bug后，使用 playwrite 进行验证，在后台或者 headless 启动。
- 计划文件都写到 /docs/plans 下面，文件命名以日期时间开头

# 产品需求/产品方案设计
- 做方案设计时，要把方案写入到 docs 文件夹下
- 保持需求文档实时更新，每次修改要在对应需求文档上面更新进度和完成度。
- 进行产品需求设计时，必须生成需求的 user story 文件，放在 user_stories目录下
- 不要受到当前已有其他页面的设计所影响，总是探索实现符合需求的最美观、最好用户体验的 UI/UX 设计方案。


# 用户故事
- 描述用户想要达到一个什么目的，如何通过产品功能一步一步实现目标。包含具体的页面操作流程，点击什么按钮，输入什么内容，查看什么界面，进行什么交互，前后端系统的预期行为等等。
- 用户故事只关注解决用户需求，改善用户体验，不要涉及技术实现细节。

# 设计规范
- 确保 UI/UX 界面简洁易用，重点突出，复杂/很少用到的功能都进行折叠隐藏，放到二级交互逻辑中。
- 基于 ui-ux-pro-max SKILL 进行网页设计
- 单纯关注设计，不要考虑技术实现。(html,css，注意这是纯设计，不是react实现)
- 只关注易用性，功能性，不要考虑前后端技术栈实现，也不要讨论后续技术实现计划和任务方案。
- 做页面设计时，不要读取任何前端后端代码文件，避免被当前实现方案所干扰。
- 任何时候都不要绘制svg，需要 icon 就是直接使用 Font Awesome icons

# 编码
- 编写代码要随时考虑到可测试性
- 代码要尽量拆分成可复用的函数，合理拆分成多个独立文件，避免面条代码，避免一个文件代码过多。
- api 路由要合理拆分成多个文件。
- **常量和配置必须提取到统一配置文件**：不要在多个文件中重复写死相同的字符串或配置值（如 token、URL、魔法数字等），应提取到 `src/worker/config.ts` 或其他统一配置文件中，然后在各处引用。
- **适度抽象，解决一类问题而非一个问题**：
  - 实现功能时先思考：这属于哪一类问题？有没有更通用的解法？
  - 数据结构和接口设计要能容纳同类场景的变化（如解析器支持多种格式，而非只支持当前用到的一种）
  - 当逻辑可以通过配置/参数来适配不同场景时，优先用配置而非硬编码
  - **平衡原则**：如果通用方案不会增加实现复杂度，就采用通用方案；如果需要大量额外代码或预测未来需求，就保持简单直接


# 日志
- 打印前端日志时，要先打印成字符串，便于复制。然后再打印对象，便于查看。

# 组件
- 尽量提取公共组件
- 尽可能把逻辑拆分成独立函数，独立组件。每个文件只应该包含一个 React 组件。
- 尽量把业务逻辑和页面渲染分离开，便于测试
- **必须把状态组件和纯渲染组件分离成两个组件**：状态组件负责管理状态和逻辑，纯渲染组件只负责 UI 展示，接收 props
- **页面各部分要尽量创建独立组件**，提高语义可读性和可复用性（如 Header、Sidebar、Content、Footer 等）
- **尽量拆分使用命名组件，避免大段 HTML**：将复杂的 JSX 拆分成多个语义化的命名组件，提高代码可读性和可维护性
- **组件库优先级**：优先使用 TanStack 生态组件（如 TanStack Table、TanStack Router、TanStack Query 等），只有在 TanStack 没有符合需求的组件时，才使用 shadcn/ui
- 添加shadcn 组件时使用命令: pnpm dlx shadcn@latest add button
